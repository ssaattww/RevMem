## レビュー観点

- T103のworkspace context・file ID・非Git repository ID生成に範囲が限定されていること
- 同一workspace/fileがprocessまたはVS Code再起動をまたいで同じIDになること
- 異なるworkspace root、remote authority、relative pathが誤って同じIDにならないこと
- Windows、POSIX、remote URIで正規化規則が明示されていること
- URI構成要素の連結が曖昧にならないこと
- workspace外documentやroot escapeを受理しないこと
- POSIXで有効なpathをWindowsまたはURI判定で過剰拒否しないこと
- application層がVS Code APIやNode runtime APIへ直接依存しないこと
- hash用途がdomain-separatedであること
- 入力objectを変更しないこと
- Red/Greenと失敗時診断artifactが確認できること

## 確認結果

### Identityの安定性と分離

- repository IDとworkspace IDはcanonical workspace URIを固定入力としてSHA-256化している
- workspace context IDはrepository IDからdomain-separatedで生成している
- file IDはrepository IDとnormalized relative pathから生成している
- serviceは時刻、乱数、process固有値、保存済み状態へ依存しない
- 同一入力を別service instanceへ渡すfixtureで完全一致を確認した
- 異なるworkspace rootと異なるremote authorityでrepository、context、file IDが分離される

### URI・path正規化

- schemeとauthorityは小文字化される
- path separatorを`/`へ統一し、`.`と`..`を解決する
- Windows file URIはdrive/path/relative pathのcase variationを同一視する
- POSIX pathはcase-sensitiveのまま保持される
- document URIのscheme・authorityがworkspaceと異なる場合は失敗する
- document pathはworkspace pathのsegment境界内であることを要求する
- caller supplied relative pathとdocument URIから導出したpathが一致しない場合は失敗する

### Layer境界

- application serviceはplatform-neutralな`ResourceUri`と`StableHash` contractだけを使用している
- Nodeの`createHash`はadapters層へ隔離されている
- adaptersからapplicationへの依存は既存の許可方向に一致する
- core層へplatform dependencyを追加していない

- 初回Red run `29979840566`は未実装moduleの`TS2307`で失敗した
- failure artifact `ci-failure-diagnostics-29979840566-1`の生成を確認した

- Green run `29980049390`は全工程成功した

## Findingと対応

### 解決済み: URI path/query/fragmentの境界衝突

- 重要度: blocking
- 初期実装はcanonical URIをraw path、`?`、raw query、`#`、raw fragmentの連結で生成していた
- pathに`?`が含まれる入力とqueryに`?`が含まれる別入力が同じ文字列を生成し、repository IDを衝突させ得た
- 再現テストを先に追加し、Actions run `29980208887`の失敗を確認した
- path segment、query、fragmentを独立してpercent-encodeする修正で解消した
- 修正後Actions run `29980296718`が全工程成功した

### 解決済み: POSIXのコロン付きファイル名を過剰拒否

- 重要度: blocking
- 初期実装はrelative path先頭がURI scheme風の場合を一律拒否し、POSIXで有効な`schema:v1.json`を扱えなかった
- 再現テストを先に追加し、Actions run `29980420185`の失敗を確認した
- Windows drive absolute形式の拒否は維持し、通常のPOSIX relative path内のcolonを許可した
- 修正後Actions run `29980479705`が全工程成功した
