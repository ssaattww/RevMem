- GitHub API、GitHub認証、ネットワーク障害からローカルGit機能が独立していること
- shell command文字列を組み立てず、Gitを引数配列で実行すること

- repository root、remote、Repository IDが決定的であること
- originを優先し、fork remoteを別Repository IDにすること
- branch完全ref、detached HEAD、HEAD commitを区別すること
- merge-baseとobject有無をGitのexit codeに従って扱うこと
- credentialや既定portの差で同一remoteが分裂しないこと
- local path remoteとUNC remoteが衝突しないこと
- Git引数へoption injectionを許さないこと

- 既存CIの失敗時診断artifactがRed原因を保持すること

## 確認結果

### 実行境界

- `NodeGitCommandExecutor`はNode.jsの`execFile`を使用する
- executableとarguments listを別々に渡し、`shell: false`を明示する
- command文字列をshellへ渡す実装はない
- stdout、stderr、exit codeを失わず、通常の非zero exitをadapter側で分類できる
- 実行ファイル不在だけを専用errorへ変換する

### Git可否とrepository discovery

- `git --version`はcwdなしで先に実行されるため、workspace pathの状態とGit可否を分離する
- `rev-parse --show-toplevel`失敗はGit利用可能な非repositoryとして返す
- root取得後のGit commandはrootをcwdとして実行する
- remoteなし、remoteあり、Git未導入、非Gitの各fixtureが独立している

### RemoteとRepository ID

- remote名は`origin`優先、その他は昇順で決定する
- SCP、SSH、HTTP、HTTPS、Git protocolのhost/pathが同じ形へ正規化される
- credential、query、fragment、`.git`がidentityへ残らない
- GitHub hostとowner/repository casingが同一視される
- SSH 22、Git 9418、HTTP 80、HTTPS 443が省略される
- file URLはpathを保持し、UNC authorityも保持する
- remoteなしroot IDはcanonical file URIのSHA-256で、同一rootでは安定し別rootでは分離する
- originがupstreamからcontributor forkへ変わるとRepository IDも変わる

### Branchとrevision

- attached branchでは`refs/heads/...`の完全refを保持する
- `symbolic-ref --quiet HEAD`のexit 1だけをdetachedとして扱う
- HEADはcommitへpeelして取得する
- merge-baseなしのexit 1だけを未取得として扱い、それ以外の異常終了を隠さない
- object存在確認は`cat-file -e`を使用する
- revisionが`-`で始まる、改行またはNULを含む場合はprocess起動前に拒否する

## Findingと対応

### 解決済み: protocol既定portによるRepository ID分裂

- 重要度: blocking
- 初期実装では`ssh://github.com:22/...`のhostを`github.com:22`として保持した
- SCP形式またはport省略SSH URLと別Repository IDになった
- 失敗する回帰テストを追加して再現した
- protocolごとの既定portを省略して解消した

### 解決済み: UNC remote authority欠落

- 重要度: blocking
- 初期実装では`file://BuildServer/Share/...`を`file:///Share/...`へ変換した
- 別server上の同一share/pathが衝突し得た
- 失敗する回帰テストを追加して再現した
- file URLのauthorityを小文字化して保持し解消した

### 解決済み: T103とのtest wiring競合

- 重要度: integration blocker
- T103とT202が分岐元の同じ`test:unit`行を変更したため、内容を包含しても3-way mergeでは競合した
- branchをT103 merge後のmainへ付け替え、T202差分を小分けに再適用した
- T103のworkspace identity testは`test:unit`に維持した
- T202 testは`test:git`へ接続し、専用`test:t202`も維持した
