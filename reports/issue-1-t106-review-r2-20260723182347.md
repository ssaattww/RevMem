## 指摘事項

- 指摘要約または「指摘なし」:
  - **[P1][blocking normal-path] 装飾の読込が一方だけ不確実な状態を両レイヤーから削除する** — `src/extension.ts:148`
    - `loadDecorations`は表示専用モデルへ状態を渡す前に、command用の`WorkspaceReviewStateSessionProvider.open`を呼ぶ。同providerは`src/adapters/workspace-review-state/workspace-review-state-session-provider.ts:262-290`でContextまたはGlobalのどちらか一方のfileがstaleなら、両方のfile entryを削除して永続化する。
    - inline再現では、Contextのrevision/path/lineCount/hash/intervalをすべてcurrent、Globalのpathだけstaleにした結果、返却stateと保存stateのContext/Global fileが両方空になった。したがって`normal-editor-decoration-global-isolation.test.ts`が示すレイヤー個別確実性は実際のExtension経路では成立せず、Globalだけの不整合で確実なContext装飾と保存済み確認状態まで失う。
    - 表示読込には非破壊のread-only loaderを使い、Context/Globalを別々に検証してからモデルへ渡す必要がある。少なくともcomposition-level回帰テストで、一方だけstaleなpersisted stateが他方を消さないことを証明する必要がある。
  - **[P2][blocking normal-path] command成功後に同じdocumentの別visible editorが更新されない** — `src/extension.ts:260-267`
    - 成功後は操作元`TextEditor`だけを`refreshEditor`している。同じdocumentを複数groupへsplit表示している場合、他方も同じstateのconsumerだが再描画されず、次のactive/visible/settings eventまで古い装飾を残す。T106終了条件の「状態更新後」のrefreshとvisible editor全体の同期を満たさない。
    - command成功後は少なくとも同一documentの全visible editor、または`refreshVisibleEditors`をrefreshする必要がある。
  - **[P2][blocking evidence gap] Extension Host testがAC-02の装飾を一切assertしていない** — `test/vscode/suite/index.ts:111-114`
    - testはmark command実行後にeditorが`visibleTextEditors`へ含まれることだけを確認しており、背景・gutter・overview ruler・hover、diff editorの空装飾、command後refreshのいずれも観測しない。このため上記production composition bugを検出できず、reportの「Extension Hostで装飾接続を検証」は証跡より強い表現になっている。
    - platform hostを観測可能にするcomposition test seam、または同等のExtension Host/manual visual evidenceが必要である。どの証跡方式を採るかについてuser confirmationが必要な契約変更はない。
  - **[P2][held concern] URI単位のdiff判定は同一URIを通常editorとdiff側へ同時表示するとdiff側も通常扱いする** — `src/extension.ts:37-42`
    - 判定は対象`TextEditor`とtabの対応を識別せず、「どこかのactiveな`TabInputText`が同じdocument URIを持つか」だけを見る。そのため同じmodified URIを通常tabとdiff tabの両方で表示すると、diff側のeditorも`false`となり装飾load対象になり得る。
    - diff単独表示の通常経路は空装飾になり、同時表示を避ければ利用可能なためheldとする。ただし「diff editorは空装飾」の絶対条件を完全には満たしていない。
  - **[P2][held concern] Global-only hoverがGlobalをContextとして表示する** — `src/application/editor-decoration/normal-editor-decoration-model.ts:212-218`
    - Global-only範囲は`contextLabel: "Global"`となり、adapterは`Context: Global`を表示する。設計4.2/4.3ではreview contextとGlobalは別概念で、11.6のhoverもContextとGlobal statusを別行にするため、将来Global-onlyが通常発生するT205/T501以降では意味が曖昧になる。
    - 現在のT105 workspace操作はContext/Globalを同時更新し、単独利用のAC-02表示は成立するためheldとする。T501接続前にsource/current-contextの表示契約を明確化する必要がある。
  - coding standards: 新規export APIのJSDoc、依存方向、手書きsource layoutにblocking違反なし。C#の`[Fact]`/`[Theory]`規則は該当なし。private test doubleの説明追加余地はあるが契約誤認を生じないためfinding化しない。
  - user-confirmation-required capability gap: なし。blocking項目はいずれもT106の既定契約内で修正可能。
