## 指摘事項

- **高・blocking normal-path** — `src/core/git-diff/git-diff-interval-mapping.ts:97-174`は`---`/`+++`まで存在してhunk直前で切れたmodified-file diffを、hunk 0件の正常fileとして受理する。`src/core/git-diff/revision-interval-mapper.ts:53-58,68-115`はそのfileを見つけると全intervalをそのまま返す。probeでは`index`・`---`・`+++`後でtruncatedしたdiffに対して`reviewed: [0,4)`、`invalidatedOld: []`となった。設計18.2の「diff parse失敗は対象fileを未確認」とAC-24に反し、不完全取得を成功扱いして古い確認済み表示を残すため、modified headerとhunkの状態機械・終端検証が必要
- **高・blocking normal-path** — `src/core/git-diff/git-diff-interval-mapping.ts:221-300`は`mapReviewedIntervalsAcrossDiff`をexportする別実装を残し、`src/core/git-diff/index.ts:10`が公開する`revision-interval-mapper.ts`版と挙動が異なる。direct module版はpure additionでcursorを挿入境界へ進めないため、`[0,4)`に2行追加したprobeでfacade版は`[0,2), [4,6)`、direct版は追加行を含む`[0,6)`を確認済みにした。exportされた同名APIの一方がAC-07/AC-08と自身の「without guessing changed lines」契約を破るため、単一実装へ統合しdirect importでも挙動を一意にする必要がある
- **高・blocking normal-path** — `src/core/git-diff/revision-interval-mapper.ts:11-31`のwhitespace比較はhunk lineから全`\s`を除くだけで、EOL変更ではremoved/added本文が同一になる点を考慮しない。`ignoreWhitespaceChanges: true`、`ignoreEolChanges: false`、`oldText: "one\r\ntwo\r\n"`、`newText: "one\ntwo\n"`のprobeで`[0,2)`を全維持した。T201 contractと設計9.6はwhitespace無視とEOL無視を独立設定にしており、EOL無視を選んでいない変更を確認済みにするため、old/new textを含むhorizontal-whitespace限定の同値性証明が必要
- **中・blocking normal-path** — `src/core/git-diff/revision-interval-mapper.ts:60-64`はEOL正規化後の文書全体が完全一致する場合しか`ignoreEolChanges`を有効にしないため、設計9.6が明記するファイル末尾改行の追加・削除を無視できない。実Gitと同じ`@@ -1 +1 @@`・no-newline markerのdiff、`oldText: "a"`、`newText: "a\n"`、`ignoreEolChanges: true`のprobeで行0がinvalidatedされた。T201の末尾改行1個だけを無視し追加空行は無視しないcontractと同じ境界判定が必要
- **中・blocking conservative-failure** — `src/core/git-diff/git-diff-interval-mapping.ts:74-84`は各sideの単調性・非重複しか検査せず、`newStart === oldStart + accumulatedDelta`のrevision間座標整合を検査しない。mapperも`newStart`を使用しないため、`@@ -2 +99 @@`を受理して99行目ではなく旧位置付近をmappingした。さらに`test/unit/git-diff-interval-mapping.test.ts:26,41,88`は実Gitが出す`+8,0`/`+6,0`ではなくoff-by-oneの`+9,0`/`+7,0`を期待し、malformed fixtureを正常系にしている。0-count座標、safe integer、prefix/gap deltaを検証して不整合はparse failureへ倒す必要がある
- **中・user-confirmation-required capability gap（T203受け入れ上はblocking）** — `src/core/git-diff/git-diff-interval-mapping.ts:59-68,120-126`はquoted pathをunquote/C-style decodeせず、先頭が`"`なので`a/`・`b/`も除去しない。実Git probeで非ASCII pathは`"a/.../\346\227..."`形式になり、parser結果もquoteとescapeを含むため通常のrepository-relative `oldPath`/`newPath`ではfileを見つけられない。該当pathの利用者は慎重な操作でもrevision追従できず、実Git構文対応というT203 scope内で修正するか、対応を延期するならユーザー確認が必要
- **中・blocking standards gate** — `src/core/git-diff/git-diff-interval-mapping.ts:4-42`の公開6 interfaceと全propertyにJSDocがなく、`parseZeroContextGitDiff`と両mapperの短いsummaryにも座標規約、`@param`、`@returns`、parse/validation failureの`@throws`がない。T201の公開contractはinterface・property・functionを文書化しており、source documentation policyのpublic/internal API contract基準を満たさない。Node test名は検証内容を説明しており、test descriptionとfolder/file layoutには追加findingなし
